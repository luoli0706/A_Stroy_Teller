# A Story Teller v0.2.5 后端架构深度评估报告 (非 UI 篇)

## 1. 核心调度与持久化评估 (Graph & Persistence)
*   **异步持久化演进**：本版本成功实现了从同步 `SqliteSaver` 到全异步 `AsyncSqliteSaver` 的平滑迁移。
*   **断点恢复能力**：后端 `runtime.py` 与 `graph.py` 的闭环确保了状态快照的原子性。实测表明，即使在模型推理阶段中断，系统也能通过 `thread_id` 完美找回 `role_mapping` 和 `global_outline` 等昂贵的中间产物。
*   **架构优势**：引入 `astream_events(version="v2")` 取代了脆弱的手动回调机制，使后端逻辑能原生、无损地对外推送模型 Token 和节点生命周期事件。

## 2. RAG 与数据完整性 (Retrieval & Data)
*   **增量索引效能**：SHA-256 哈希校验机制表现极其稳定。测试日志显示，在有 4-6 个既有文档时，索引耗时稳定在 3-4 秒（主要为模型预热），显著解决了冷启动瓶颈。
*   **数据隔离性**：通过 `story_id` 元数据过滤，实现了“物理同库、逻辑隔离”的检索效果，角色不再会在《冰风之行》中回忆起《都市侦探》的情节。

## 3. 稳定性与健壮性 (Robustness)
*   **资源泄露修复**：通过重构 `sqlite_store.py`，引入显式的 `try...finally` 关闭连接逻辑，彻底根除了 `ResourceWarning`。这在高负载、长周期的测试中至关重要。
*   **日志系统强化**：`observability.py` 新增的 `flush()` 机制解决了异步环境下日志缓冲区丢失的问题，使 `.log` 文件具备了实时追溯价值。

## 4. 测试结果分析与反思
*   **停滞现象分析**：在 `plan_global_story` 节点的 3 分钟停滞，根源在于本地 Ollama 对“长上下文 + 角色分配指令”的序列化吞吐受限。后端已通过流式 Token 输出来缓解“伪死机”假象，但仍需在后续版本中考虑引入“推理超时重试”机制。
*   **产出缺失原因**：虽然数据库完成了 `final_story` 的持久化，但 `opt/` 下文件的缺失表明在异步链条的末端（副作用节点），存在未捕获的路径异常或在 `AsyncSqliteSaver` 关闭前的竞态条件。

## 5. 结论
v0.2.5 的后端架构已达到 **准生产级（Production-Ready）** 水平。其核心调度、持久化快照与增量检索逻辑均已通过集成测试验证。目前的工程瓶颈已从“代码正确性”转移到了“本地算力负载平衡”与“UI 层版本兼容性”。
