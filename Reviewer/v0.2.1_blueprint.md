# A Story Teller v0.2.1 版本蓝图 - 角色扮演与通用设定

## 1. 版本目标
本版本旨在解耦角色设定与特定故事的关系，使角色成为具备通用属性（Physical & Personality Traits）的“演员”，能够动态入驻不同的故事框架。

## 2. 核心任务清单

### 2.1 角色设定通用化 (Generic Role Profiles)
- [ ] **剥离专有背景**：修改 `role/` 下所有角色的 `profile.md`，移除与特定故事相关的身份（如“时钟塔调查员”），替换为通用属性：
    - 基本信息：身高、体重、外貌风格。
    - 性格特质：核心性格、待人接物方式、禁忌、癖好。
    - 语言风格：说话的语气、常用口头禅。
- [ ] **维护长期记忆**：角色的长期记忆（Memory）应保留跨故事的成长，但叙事时需根据当前故事背景进行检索过滤。

### 2.2 故事框架角色槽位 (Story Framework Slots)
- [ ] **定义 Slot 接口**：在 `stories/{story_id}/framework.md` 中新增 `## Role Slots` 部分，定义故事所需角色的原型（如：侦探、助手、嫌疑人）。
- [ ] **动态替换逻辑**：在 `plan_global_story` 节点，由 LLM 根据当前“演员”的通用属性，自动将其分配到合适的“故事槽位”。演员的名字将替换槽位中的占位名。

### 2.3 溢出角色处理 (Overflow Roles)
- [ ] **新增角色机制**：如果参与生成的演员数量超过了框架定义的 Slot 数量，系统将允许演员以“独立第三方”或“新访客”身份强行入驻，LLM 需自动为其在剧情中创造合理切入点。

## 3. 技术路线
1. **Prompt 改造**：在 `generate_role_view` 的 Prompt 中明确区分“演员（Actor）”与“角色槽位（Slot）”。
2. **逻辑分层**：引入 `app/role_mapping.py` 处理演员与槽位的匹配度评估（基于 LLM）。

## 4. 后续评估计划
- **演员适配度评估**：评估不同性格的演员扮演同一槽位时，故事走向的多样性。
- **一致性审计**：确保演员的通用性格在不同故事框架下保持稳定，不被故事背景完全“洗脑”。
